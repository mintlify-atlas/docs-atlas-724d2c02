## What is a deployment lock?

A deployment lock prevents multiple deployments from running at the same time. When a lock is held, only the owner of that lock can trigger deployments to the locked environment. This is critical for:

- **Coordinated releases** — Ensure only one person deploys to production at a time
- **Maintenance windows** — Block all deployments during planned downtime
- **Emergency holds** — Quickly freeze deployments when an incident is in progress
- **CI gate integration** — Automatically pause deployments when tests are failing

## How it works

The Lock Action stores lock state as a JSON file on a dedicated Git branch in your repository. No external infrastructure is required — locks live entirely within GitHub.

<Steps>
  <Step title="Add the Action to your workflow">
    Reference `github/lock@vX.X.X` in your workflow file and configure the desired mode and environment.
  </Step>
  <Step title="Trigger a lock">
    Comment `.lock` on a pull request (IssueOps mode) or dispatch a workflow with `mode: lock` (headless mode).
  </Step>
  <Step title="Deploy with confidence">
    Only the lock owner can deploy while the lock is active. Others will see a clear message explaining who holds the lock.
  </Step>
  <Step title="Release the lock">
    Comment `.unlock` on the PR or dispatch a workflow with `mode: unlock` to release the lock and allow others to deploy.
  </Step>
</Steps>
