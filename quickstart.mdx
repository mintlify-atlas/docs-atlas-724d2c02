## Prerequisites

- A GitHub repository where you have permission to create workflow files
- The repository must allow GitHub Actions to create and push to branches (check **Settings → Actions → General → Workflow permissions** and enable "Read and write permissions")

## IssueOps setup

This example wires up `.lock` and `.unlock` as PR comment triggers. Any collaborator with write access to the repository can claim or release the deployment lock by commenting on a pull request.

<Steps>
  <Step title="Create the workflow file">
    Create `.github/workflows/lock-or-unlock.yml` in your repository:

    ```yaml
    name: lock-or-unlock

    on:
      issue_comment:
        types: [created]

    permissions:
      pull-requests: write
      contents: write

    jobs:
      lock-or-unlock:
        if: ${{ github.event.issue.pull_request }}
        runs-on: ubuntu-latest
        steps:
          - uses: github/lock@vX.X.X
            id: lock
    ```

    The `if` condition restricts the job to comments on pull requests only — issue comments on regular issues are ignored.
  </Step>

  <Step title="Open a pull request">
    Push a branch and open a pull request. The workflow will not trigger until a comment is posted.
  </Step>

  <Step title="Claim the lock">
    Comment `.lock` on the pull request. Within seconds:

    - The Action adds an 👀 reaction to your comment to confirm it was detected
    - The lock is written to a `production-branch-deploy-lock` Git branch in your repository
    - A sticky comment is posted to the PR:

    ```
    ### 🔒 Deployment Lock Claimed

    You are now the only user that can trigger deployments to the `production`
    environment until the deployment lock is removed

    > This lock is _sticky_ and will persist until someone runs `.unlock production`
    ```

    <Note>
    The lock is environment-scoped. The default environment is `production`. To lock a different environment, comment `.lock staging` or `.lock development`.
    </Note>
  </Step>

  <Step title="Verify the lock">
    Comment `.wcid` ("where can I deploy?") on any PR to check the current lock state. The Action will reply with the lock details:

    ```
    ### Lock Details 🔒

    The deployment lock is currently claimed by __octocat__

    - __Reason__: `null`
    - __Branch__: `fix/critical-bug`
    - __Created At__: `2024-03-15T10:30:00.000Z`
    - __Created By__: `octocat`
    - __Sticky__: `true`
    - __Environment__: `production`

    The current lock has been active for `0d:2h:15m:30s`
    ```
  </Step>

  <Step title="Release the lock">
    Comment `.unlock` on any pull request to release the lock. The `production-branch-deploy-lock` branch is deleted and deployments are unblocked.

    <Tip>
    To add a reason when claiming the lock, append `--reason` to your comment: `.lock --reason deploying hotfix for #1234`
    </Tip>
  </Step>
</Steps>

## Locking a specific environment

By default the Action targets the `production` environment. To support multiple environments, configure `environment_targets`:

```yaml
- uses: github/lock@vX.X.X
  id: lock
  with:
    environment_targets: "production,staging,development"
```

Commenters can then target a specific environment:

| Comment | Effect |
| ------- | ------ |
| `.lock` | Lock the default environment (`production`) |
| `.lock staging` | Lock the `staging` environment |
| `.lock development` | Lock the `development` environment |
| `.unlock staging` | Unlock the `staging` environment |
| `.lock --global` | Lock all environments simultaneously |

## Headless setup (workflow dispatch)

For programmatic locking — from scheduled workflows, conditional logic, or a manual UI trigger — use headless mode by setting the `mode` input.

<Tabs>
  <Tab title="Lock via workflow dispatch">
    ```yaml
    name: lock-dispatch

    on:
      workflow_dispatch:
        inputs:
          reason:
            description: "Reason for claiming the deployment lock"
            required: false
          environment:
            description: "Environment to lock (production, staging, etc.)"
            required: true
            default: "production"
          mode:
            description: "Operation: lock, unlock, or check"
            required: true
            default: "lock"

    permissions:
      contents: write

    jobs:
      lock:
        runs-on: ubuntu-latest
        steps:
          - uses: github/lock@vX.X.X
            id: lock
            with:
              mode: ${{ github.event.inputs.mode }}
              reason: ${{ github.event.inputs.reason }}
              environment: ${{ github.event.inputs.environment }}
    ```

    Trigger this workflow from the GitHub Actions UI (**Actions → lock-dispatch → Run workflow**) and fill in the inputs. No pull request or comment is required.
  </Tab>
  <Tab title="Check lock status">
    ```yaml
    name: check-lock

    on:
      workflow_dispatch:

    permissions:
      contents: read

    jobs:
      check:
        runs-on: ubuntu-latest
        steps:
          - uses: github/lock@vX.X.X
            id: lock
            with:
              mode: check

          - name: Print lock status
            run: |
              echo "Locked: ${{ steps.lock.outputs.locked }}"
              echo "Locked by: ${{ steps.lock.outputs.created_by }}"
              echo "Lock branch: ${{ steps.lock.outputs.branch }}"
    ```

    The `check` mode only reads the lock file and does not modify any state. It requires only `contents: read`.
  </Tab>
  <Tab title="Conditional lock">
    ```yaml
    name: lock-on-failure

    on:
      workflow_run:
        workflows: ["test"]
        types: [completed]

    permissions:
      contents: write

    jobs:
      lock-if-failing:
        if: ${{ github.event.workflow_run.conclusion == 'failure' }}
        runs-on: ubuntu-latest
        steps:
          - uses: github/lock@vX.X.X
            id: lock
            with:
              mode: lock
              reason: "Automated lock: test suite is failing"
              environment: production
    ```

    This workflow automatically claims the production lock whenever the test suite fails, preventing deployments until the lock is manually released.
  </Tab>
</Tabs>

## Action outputs

After the Action runs, you can reference its outputs in subsequent steps:

```yaml
- uses: github/lock@vX.X.X
  id: lock
  with:
    mode: check

- name: Gate on lock status
  if: ${{ steps.lock.outputs.locked == 'true' }}
  run: |
    echo "Deployment is locked by ${{ steps.lock.outputs.created_by }}"
    echo "Reason: ${{ steps.lock.outputs.reason }}"
    exit 1
```

Key outputs:

| Output | Description |
| ------ | ----------- |
| `triggered` | `"true"` if an IssueOps trigger was matched |
| `headless` | `"true"` if the Action ran in headless mode |
| `locked` | `"true"` if a lock exists (only set in `check` mode) |
| `created_by` | GitHub username of the lock owner (`check` mode) |
| `created_at` | ISO 8601 timestamp when the lock was claimed (`check` mode) |
| `reason` | Reason provided when the lock was claimed (`check` mode) |
| `branch` | Branch that holds the lock (`check` mode) |
| `global_lock_claimed` | `"true"` if a global lock was just claimed |
| `global_lock_released` | `"true"` if a global lock was just released |

<Warning>
The `locked`, `created_by`, `created_at`, `reason`, and `branch` outputs are only populated when `mode` is set to `check`. In lock and unlock operations these outputs are empty.
</Warning>
